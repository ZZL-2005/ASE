你现在的任务不是继续开发新功能，而是对当前 ASE 项目进行**受控自测**、**问题定位**、**文档化验证**。

## 0. 总目标

请你对 `/new_disk4/jiayue_pu/ZZL/ASE` 当前实现进行一次系统化测试，验证以下内容是否真实成立，而不是仅凭代码结构推断：

1. 本地服务是否确实可启动
2. Rocket.Chat / Email 是否确实可通信
3. agent / user / channel 闭环是否真实打通

7. 当前文档是否与实际实现一致

你的目标不是“证明它能用”，而是**诚实地找出不能用、没测到、测不实的地方**。

## 1. 绝对约束（必须遵守）

### 1.1 开发与测试目录限制
你只能在以下目录内进行工作：

- `/new_disk4/jiayue_pu/ZZL/ASE`

不得直接修改以下仓库：

- `/new_disk4/jiayue_pu/ZZL/nanobot`
- `/new_disk4/jiayue_pu/ZZL/OpenSandbox`

如需参考代码，只能读取，不能修改。

### 1.2 进程与端口安全限制
**严禁清理、杀死、停止、重启其他用户的进程或占用端口。**

具体要求：

- 不得使用 `kill`, `pkill`, `killall` 去处理不明确归属的进程
- 不得为了抢占端口而终止未知进程
- 如果发现端口被占用，只能：
  1. 识别占用情况
  2. 记录问题
  3. 修改 ASE 自己的测试端口或配置
  4. 或明确提示“因端口冲突无法继续该测试”

**禁止基于猜测清理环境。**

### 1.3 不得假设成功
禁止以下行为：

- 没实际运行就声称“已完成测试”
- 没真实收发消息就声称“通信正常”
- 没真实生成轨迹文件就声称“trajectory 正常”
- 没真实执行 replay 就声称“支持恢复”
- 没阅读实现和日志就凭 README 猜测系统行为

### 1.4 版本与环境要求
测试过程中如果涉及依赖、镜像、服务版本，请记录：

- 实际使用的镜像 tag
- 实际使用的包版本
- docker compose / python / node 等关键版本
- 关键服务监听端口
